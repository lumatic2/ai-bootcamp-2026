# Solution 발산·수렴 — Problem A 확정 후

> 작성: 2026-07-15 · Problem A 확정: **"온라인에서 처음 보는 브랜드 옷을 살 때, 사이즈를 못 믿어 도박하듯 주문하고 반품한다"**
> 페르소나: 온라인에서 처음 보는 브랜드 옷을 사는 20대 중후반 남성. 자기 실측 치수 모름, 사이즈 실패·반품 경험 있음.

## 1. 발산 — solution 후보 12개 (평가 금지 단계의 산출, 순서 무의미)

## 4. 수렴 결정 — 추천

**Top 1: S1 사이즈 번역기, S2·S9를 컴포넌트로 흡수한 형태.**

S1과 S2가 동점(21)인데 별개 제품이 아니라 **같은 제품의 엔진 두 층**이다: S2(리뷰 핏 마이닝)는 S1의 근거 생성기이고, S9(사이즈표 파싱)는 S1의 입력 확장이다. 따라서:

> **MVP 계약**: 사용자가 ① 잘 맞는 브랜드+사이즈를 고르고 ② 사려는 상품(브랜드 선택 + 선택적으로 리뷰 텍스트/사이즈표 붙여넣기)을 주면 → **추천 사이즈 + 신뢰도 + 근거 2종(실측 비교, 리뷰 인용)**을 돌려준다.

**Thin slice 순서 (내일 구현, 앞이 죽어도 뒤가 성립하도록)**:
1. **Slice 1 (코어, 오전)**: 큐레이션 실측표 DB(무신사 실측 페이지에서 10~20개 브랜드 × 대표 카테고리 1개=상의) + 관계 매핑 로직 + 결과 화면. — 이것만으로 데모 성립
2. **Slice 2 (AI 근거, 오후 초)**: 리뷰 텍스트 붙여넣기 → LLM 핏 신호 추출 → 근거 카드. — 실측표가 없는 브랜드 커버
3. **Slice 3 (여유 시)**: 사이즈표 스크린샷 업로드 파싱(S9) / "도박 비용" 공유 카드(S5)

**화면 3개**: 입력(브랜드+사이즈 선택) → 대상 상품 입력 → 결과(추천+신뢰도+근거). GA 이벤트 6종은 이 3화면 전환에 그대로 심는다 (`start_input`→`view_result`→`adopt_size`/`repeat_query`).

**API**: `/api/translate` (실측 매핑+종합), `/api/mine-reviews` (LLM 핏 신호 추출). 키는 `.env.local` 준비 완료.

**포지셔닝 문장 (발표용)**: "True Fit이 B2B로 증명한 '사이즈는 관계다' 접근을, 한국에서 처음으로 소비자가 직접 쓰는 형태로 검증한다. 실측 몰라도 되고, 근거(리뷰 원문)를 보여준다."
