# Day 2 오전 발표 스크립트 — 결정과정·리서치·피벗·현재 상태 (2026-07-16)

> 공식 평가항목 매핑: `docs/presentation-2026-07-15.md` §⭐ — 과정(문제정의 30 + 검증설계 30 + 학습·피벗 20) = 80%, 구현 = 10%.

## 발표 순서 (7파트, ~10분 기준)

### 1. 원안과 1차 피드백 — 출발점
- 원안: 20~30대 남성 디지털 옷장 (사놓은 옷 까먹지 않게 + 코디 추천).
- 1차 피드백: **페르소나 불명확 · 흔한 주제 · 문제정의 고민 부족.**
- "피드백을 반박하지 않고, **리서치로 검증해보기로 했다**" ← 여기가 스토리 시동.

### 2. 리서치 방법 — 품질을 어떻게 관리했나 (득점 포인트 ①)
- **5경로 병렬 채굴** (평가 금지 발산): 소비자 불만 커뮤니티 / 경쟁 지형 / 공식 통계 / 공급자 pain / 학술 논문 → 후보 24개. (`references/research-clothes-20260715/01~05`)
- **근거 레벨 E1~E5 태깅**: 모든 사실에 출처 등급을 붙임 — E1 법령·공식통계(한국소비자원) > E2 논문·공식발표 > E3/E4 언론·실측 > E5 커뮤니티 존재 증명. 등급이 낮으면 주장 강도도 낮춘다.
- **정직성 원칙의 실례**: "온라인 의류 반품률 30~40%"는 업계 블로그발이라 **인용하지 않았다**. 대신 E1인 KCA 피해구제 건수(2022년 278건 → 2024년 592건, 사유 1위 청약철회 48.4%)로 규모를 말한다. — "공식 통계가 없다는 사실 자체를 명시하는 것이 검증"
- **적대적 존재 검증**: "세상에 없는 아이디어"가 아님을 우리가 먼저 확인 — True Fit(글로벌 B2B), Virtusize(서울 오피스, B2B 위젯), 무신사 마이사이즈·pullsize(실측 입력 기반). (`07-competitor-verification.md`)

### 3. 스코어 루브릭으로 문제 선정 (득점 포인트 ②)
- 24개 후보 → 8개 클러스터 → **6축 루브릭 채점** (강도·빈도·접근성·검증가능성·역량·근거품질, 각 1~5, 만점 30). (`06-problem-scoring.md`)
- 결과: **Problem A "처음 보는 브랜드 사이즈 도박" 27점** — 유일하게 5경로 전부에서 독립 검출 + E1~E5 전 층 근거.
- 원안(H 클러스터)은 19점, 특히 **근거 축 1점**: 국내외 7개+ 앱이 라이브인데 성공 증거도 실패 원인도 미공개 — "흔한데 안 풀린 이유를 모르는" 최악의 조합.
- 명시할 규약: "점수는 토론 입력이지 자동 의사결정기가 아니다 — tie-breaker는 팀원의 직접 경험 증언."

### 4. 피벗 — 폐기가 아니라 Learn
- 가설(디지털 옷장) → 검증(5경로 리서치) → **반증 발견** → 피벗. 강사가 요구한 루프 그 자체를 문제 선정 단계에서 이미 한 번 돌렸다.
- 연속성: 페르소나는 유지, pain만 교체. 원안의 코디·구매 추천은 버린 게 아니라 2단계 로드맵 — 너비가 모으는 "잘 맞는 브랜드+사이즈" 데이터가 그대로 체형·취향 프로필이 된다. (`00-meeting-brief.md` §0)

### 5. 솔루션 결정 — 발산·수렴과 포지셔닝
- Problem 확정 후 solution 별도 발산 → 수렴 기준 4개(AI 필수 / 하루 구현 / 당일 측정 / 충돌 목록 회피)로 **사이즈 번역기** 선택. (`08-solution-convergence.md`)
- 핵심 프레임(학술 검증): **"사이즈는 속성이 아니라 관계다"** — Amazon WWW 2018. 실측 표기가 정확해도 같은 실측=같은 핏이 아니다.
- 포지셔닝(참신함 주장 안 함): "True Fit이 접근의 타당성을 증명했고, **한국에 소비자가 직접 쓸 수 있는 형태가 비어 있다.**" 경쟁자들은 전부 '내 실측을 안다'를 전제 — 우리는 실측 없이 관계 신호로 번역.
- **AI 필수 요건**: 비정형 사이즈표 해석 + 시드에 없는 브랜드의 AI 사이즈표 검색(커스텀 앵커) — LLM 없이는 성립하지 않는 코어.

### 6. 검증 설계 — 만들기 전에 반증 조건부터 (득점 포인트 ③)
- 가설: "실측 치수 없이 관계 신호만으로 **적중률 ≥60%**(우연 ~33%)" / **반증: <40%면 엔진 가설 기각** → 피벗하고 그 과정을 발표. (`experiments/01`)
- KPI 6종은 팔란티어 프레임(What decision / How much impact / What data)으로 정렬 — MAU·페이지뷰 안 씀. 적중률(North Star)·완료율·60초·채택률·재조회율·예측불가율. (`presentation-2026-07-15.md` §3)
- 선행 가설 E0("실측표를 3시간 내 10개+ 확보")도 실측으로 닫음: 17개 시도 → 11 수집, **GO**. 실패 6건의 원인("기준표 탭" 함정)까지 기록. (`09-size-chart-feasibility.md`)

### 7. 현재 배포 상태 + 오늘 계획
- **라이브**: https://bootcamp.askewly.com — 3스텝 플로우, **48브랜드 일괄 번역 그리드**(적합도 % + 라벨), Step1 AI 사이즈표 검색(커스텀 앵커, 출처 URL 필수·신뢰도 캡), 챗봇(function calling), GA4 이벤트 8종 계측.
- **협업 증거**: 팀원 디자인 스펙(`references/nubi-style-spec/SPEC.md`)을 그대로 채택해 전체 리스킨 / 팀원 데모 쇼핑몰과 양방향 연동 계약(`docs/mall-integration.md`) — 몰→너비 딥링크는 배포 완료.
- **오늘**: V0 팀원 교차검증(표본 ≥8, <40%면 반증 발동) → V1 블라인드 15~20명 배포·측정 → KPI 6종 실측값 → Learn 기록으로 실험 4섹션을 닫는다. "오답 사례도 그대로 보여줄 것"이 원칙.
- 비전 1장: 최종 형태는 쇼핑몰 상품 상세 위젯(True Fit이 B2B로 증명한 유통 형태) — 목업은 "실제 연동 아님" 구두 명시. (`references/forme-design-spec/vision-mockup-nubi-widget.png`)

## 예상 질문 대비
- **"적중률 근거가 아직 없네요?"** → "맞습니다. 그래서 오늘 오전이 V0 교차검증입니다. 반증 조건(<40%)을 어제 먼저 박아뒀고, 기각되면 기각 과정을 발표하는 것까지가 설계입니다."
- **"이미 있는 서비스 아닌가요?"** → 적대적 검증 결과로 답변 (§2 마지막): B2B 위젯 vs 소비자 독립 도구, 실측 입력 vs 관계 신호.
- **"반품률이 얼마나 되는데요?"** → "공식 통계가 없습니다"라고 먼저 말하고 KCA E1 수치 제시 — 이 절제가 우리 리서치 품질 관리의 예시.
