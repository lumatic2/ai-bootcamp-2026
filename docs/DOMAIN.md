# 도메인 규칙 — 3일 가설 검증형 제품 개발

> 팀 플레이북이 공유하는 용어와 evidence 기준. 제품 도메인 자체는 팀 문제 선정 후 별도로 추가한다.

## 도메인
불확실한 문제를 짧은 시간 안에 가설, 최소 제품, 관찰 데이터와 학습으로 연결하는 팀 제품 실험.

## 적용 범위
- 대상: AI 바이브코딩 부트캠프 팀의 문제 선정, 구현, 측정, 학습, 발표
- 제외: 아이디어의 시장성에 대한 자동 결론, 근거 없는 사용자 대변, mock 데이터를 실제 관찰로 표시하는 행위
- 기준일: 2026-07-14 작성 / 마지막 확인 2026-07-14

## 정의
- **가설**: 어떤 사용자와 문제 또는 솔루션에 관한 반증 가능한 문장.
- **Build**: 가설을 검증하는 데 필요한 최소한의 제품 또는 관찰 장치.
- **Measure**: 구현 전에 정한 metric과 수집 방법으로 관찰하는 단계.
- **Data**: 출처, 시점, 표본과 한계가 표시된 관찰값. mock은 명확히 mock으로 표시한다.
- **Learn**: 데이터가 가설을 지지·반박·판단불가 중 어디로 이동시켰는지와 다음 행동을 기록한 결론.
- **Evidence chain**: 문제 증거 → 가설 → 제품 변화 → metric → raw data → 판단 → 다음 가설의 연결.

## 자주 헷갈리는 케이스
1. 기능을 완성했지만 사용자가 보지 않음 — Build 완료일 뿐 Measure와 Learn은 미완료다.
2. 인터뷰 한 명의 긍정 반응 — 방향 신호일 수 있지만 모집 편향과 표본 한계를 함께 기록한다.
3. 발표용 숫자를 임의 생성 — mock 데이터로 표기하면 시연 자료, 실제 측정처럼 쓰면 evidence가 아니다.
