SLIDE 3 · 솔루션 선택과 가설

"사이즈는 속성이 아니라 관계다"

솔루션 발산 → 수렴 기준 4개

  • Problem 확정 후 솔루션을 별도로 발산, 4기준으로 수렴: AI 필수 / 하루 구현 / 당일 측정 가능 / 경쟁 충돌 목록 회피 → 사이즈 번역기 선택
  • 핵심 프레임(학술 근거): 같은 실측이어도 같은 핏이 아니다 — 사이즈는 사람과 옷 사이의 관계 (Amazon, WWW 2018)
  • AI 필수 요건: 비정형 사이즈표 해석 + 시드에 없는 브랜드의 AI 사이즈표 웹 검색 — LLM 없이는 성립하지 않는 코어

포지셔닝 — 참신함을 주장하지 않는다

  • "True Fit이 접근의 타당성을 B2B로 증명했고, 한국에 소비자가 직접 쓰는 형태가 비어 있다"
  • 경쟁자들은 전부 '내 실측 치수를 안다'를 전제 — 우리는 줄자 없이, 잘 맞는 옷 한 벌(관계 신호)로 번역

가설 — 만들기 전에 반증 조건부터

H1. 실측 치수 없이, 잘 맞는 브랜드+사이즈라는 관계 신호만으로 처음 보는 브랜드의 추천 사이즈를 적중률 60% 이상으로 예측할 수 있다 (우연 수준 ~33%, S/M/L 3지선다 가정)

반증 조건. 블라인드 검증 적중률 40% 미만이면 엔진 가설 기각 → 피벗하고, 기각 과정 자체를 발표한다

선행 가설 E0도 실측으로 닫음

  • "브랜드 실측표를 3시간 내 10개+ 확보할 수 있다" → 17개 시도, 11개 수집, GO
  • 실패 6건의 원인("기준표 탭" 함정 — 실측이 아닌 제품 기준치)까지 기록 — 이 발견이 이후 데이터 파이프라인 설계의 입력이 됨

말할 포인트: "KPI는 팔란티어 프레임(어떤 결정을 · 얼마나 바꾸나 · 무슨 데이터로)으로 골랐고, MAU·페이지뷰는 쓰지 않았습니다."

Learn 3개

  • ① 피벗은 손실이 아니었다 — 원안(디지털 옷장)을 리서치로 반증하고 갈아탄 것이 이번 프로젝트에서 가장 값비싼 산출물. 가설→검증→반증→피벗 루프를 제품 만들기 전에 1회전
  • ② 사용자의 조건은 단 하나, 정확도 — 설문 15명 전원이 같은 답. 기능 추가보다 적중률 검증(표본 확보)이 최우선이라는 우선순위가 데이터로 확정
  • ③ 데이터는 병목이 아니라 플라이휠이었다 — 시드 48개로 시작, 공개 실측 API 발견 + 라벨 복구 + AI 수집 축적으로 하루 만에 213 브랜드·533 상품. 사용할수록 쌓이는 구조까지 배포
